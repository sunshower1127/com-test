# HwpAutomation COM 자동화 버전별 연결 가이드

한글(HWP) COM 자동화의 핵심은 **ProgID `HWPFrame.HwpObject`가 모든 버전에서 동일**하지만, 내부 CLSID·레지스트리·API 동작은 버전마다 상이하다는 점이다. 특히 2024년을 기점으로 ActiveX(HwpCtrl.ocx)가 완전 제거되어 OLE Automation(`HWPFrame.HwpObject`)이 유일한 공식 자동화 경로가 되었다. 다중 버전이 설치된 환경에서는 마지막으로 COM 등록된 버전만 작동하며, 한글 2024에서는 .NET Interop 호환 깨짐, Visible 속성 버그, 다중 인스턴스 충돌 등 주요 breaking change가 존재한다. 이 보고서는 한글 2018~2024(NEX) 전 버전에 걸친 COM 연결 노하우를 실무 관점에서 정리한다.

---

## ProgID와 CLSID는 버전에 따라 조용히 바뀐다

한글 COM 자동화에서 사용하는 ProgID 체계는 세 단계로 구성된다. 버전 독립적인 **`HWPFrame.HwpObject`**, 그리고 버전별로 **`HWPFrame.HwpObject.1`**(한글 2020 이하)과 **`HWPFrame.HwpObject.2`**(한글 2022 이상)가 존재한다. `HWPFrame.HwpObject`의 `CurVer` 레지스트리 서브키가 가장 최근 설치된 버전의 ProgID를 가리키므로, 일반적으로 `win32.Dispatch("HWPFrame.HwpObject")`는 마지막으로 등록된 한글 버전에 연결된다.

CLSID 역시 버전에 따라 다르다. 한글 2020 이하는 **`{2291CEFF-64A1-4877-A9B4-68CFE89612D6}`**, 한글 2022 이상은 **`{2291CF00-64A1-4877-A9B4-68CFE89612D6}`**를 사용한다(첫 번째 세그먼트 마지막 바이트만 `CEFF` → `CF00`으로 변경). TypeLib GUID는 전 버전 공통으로 `{7D2B6F3C-1D95-4E0C-BF5A-5EE564186FBC}`이다.

주의해야 할 점은 "HwpAutomation"이라는 이름이 ProgID가 **아니라** 타입 라이브러리 이름이라는 것이다. 실제 COM 연결에 사용하는 ProgID는 반드시 `HWPFrame.HwpObject` 계열이어야 한다. 한글 설치 디렉터리의 `Bin` 폴더에는 두 개의 `.tlb` 파일이 존재하는데, `HwpObject.tlb`(권장)과 `HwpAutomation.tlb`이 그것이다. 한컴 공식 입장에 따르면 **HwpObjectLib 사용을 권장**하며, HwpAutomation.tlb은 부모 윈도우 없이는 제대로 동작하지 않는 제약이 있다.

레지스트리 등록 위치도 중요하다. 한글은 전통적으로 **32비트 애플리케이션**이기 때문에 64비트 Windows에서는 `WOW6432Node` 경로에 등록된다. 주요 레지스트리 경로는 다음과 같다:

```
HKCR\HWPFrame.HwpObject\CurVer → "HWPFrame.HwpObject.2"
HKCR\CLSID\{2291CF00-...}\LocalServer32 → "C:\...\Bin\hwp.exe -Automation"
HKCR\CLSID\{2291CF00-...}\TypeLib → "{7D2B6F3C-...}"
HKCU\SOFTWARE\HNC\HwpAutomation\Modules → 보안 모듈 DLL 경로
```

한글 내부 버전 번호 체계는 한글 2018=10.x, 2020=11.x, 2022=12.x, **2024=13.x(코드명 130)**이다. 설정 파일 경로도 `%appdata%\HNC\User\Common\130` 형태로 이 번호를 따른다.

## 한글 2024에서 발생한 주요 breaking change 세 가지

### Visible 속성 버그

한글 2022까지 `hwp.XHwpWindows.Item(0).Visible = True` 한 줄이면 창이 표시되었으나, **한글 2024에서는 이 호출이 동작하지 않는다**. 작업 표시줄 프리뷰에만 나타나고 실제 창은 보이지 않는 현상이 발생한다. 한컴이 확인한 워크어라운드는 False→True 토글이다:

```python
hwp.XHwpWindows.Item(0).Visible = False  # 먼저 끔
hwp.XHwpWindows.Item(0).Visible = True   # 다시 켬
```

### .NET Interop RPC 에러

한글 2024에서 C# Interop DLL을 통한 COM 호출 시 **"Server RPC 에러"**가 발생한다. `hwp.HParameterSet.HParaShape.BorderFill` 같은 중첩 프로퍼티 접근에서 특히 문제가 된다. 한컴이 확인한 공식 워크어라운드는 타입 참조 대신 **`dynamic` 키워드로 late-binding**을 사용하는 것이다:

```csharp
Type hwpType = Type.GetTypeFromProgID("HWPFrame.HwpObject.2");
dynamic hwp = Activator.CreateInstance(hwpType);
// Interop 참조 대신 dynamic으로 모든 프로퍼티 접근
```

### 다중 인스턴스 충돌

한글 2022까지는 `EnsureDispatch`를 여러 번 호출해 독립 인스턴스를 생성할 수 있었으나, **한글 2024 초기 빌드에서는 두 번째 인스턴스 생성 시 첫 번째 창이 사라지는 버그**가 있었다. 이 문제는 2025년 2월 패치로 수정되었으나, 멀티 인스턴스를 사용하는 코드는 반드시 한글 버전과 패치 수준을 확인해야 한다.

## ActiveX 종료가 가져온 생태계 지각변동

2024년 1월 1일부로 HwpCtrl.ocx(ActiveX 컨트롤) 기술지원이 공식 종료되었고, **2024년 6월 24일 패치로 한글 2018·2020·2022에서도 OCX 파일이 삭제**되었다. 한글 2024에는 애초에 OCX가 포함되지 않았다. 이로 인해 `HwpCtrl.HwpObject` ProgID를 사용하던 모든 애플리케이션이 동작을 멈췄다.

| 구분           | HwpCtrl (OCX)         | HwpAutomation (OLE)         |
| -------------- | --------------------- | --------------------------- |
| 타입           | ActiveX 컨트롤        | Out-of-process COM 서버     |
| ProgID         | `HwpCtrl.HwpObject`   | `HWPFrame.HwpObject`        |
| 보안 모듈 경로 | `HNC\HwpCtrl\Modules` | `HNC\HwpAutomation\Modules` |
| 현재 상태      | **완전 제거**         | 유일한 공식 경로            |

마이그레이션 시 가장 흔한 실수는 보안 모듈 레지스트리 경로를 `HwpCtrl\Modules`에 그대로 두는 것이다. **반드시 `HwpAutomation\Modules`로 변경**해야 보안 팝업 없이 파일 I/O가 가능하다. Python의 `win32com.client.Dispatch("HWPFrame.HwpObject")`는 원래부터 OLE Automation을 사용하므로 Python 기반 코드는 OCX 제거의 직접적 영향을 받지 않는다.

## 보안 모듈 설정은 정확한 3단계를 따라야 한다

`RegisterModule` 없이 파일을 열거나 저장하면 "한글을 이용하여 위 파일에 접근하려는 시도가 있습니다" 보안 팝업이 매번 표시된다. 정확한 설정 절차는 다음과 같다:

**1단계: DLL 확보.** 한컴디벨로퍼(developer.hancom.com/hwpautomation) 또는 GitHub(`hancom-io/devcenter-archive`)에서 `보안모듈(Automation).zip`을 다운로드하여 `FilePathCheckerModuleExample.dll`을 안전한 경로에 배치한다.

**2단계: 레지스트리 등록.** `HKCU\SOFTWARE\HNC\HwpAutomation\Modules` 경로에 문자열 값을 생성한다. 값 이름은 자유롭게 지정(예: `FilePathCheckerModule`)하고, 값 데이터에 DLL 전체 경로를 입력한다. **경로에 따옴표(`"`)가 포함되면 오류가 발생**하므로 반드시 제거해야 한다.

**3단계: 코드에서 호출.** `hwp.RegisterModule("FilePathCheckDLL", "FilePathCheckerModule")`에서 두 번째 인자는 레지스트리의 **값 이름**과 정확히 일치해야 한다. `pyhwpx` 라이브러리를 사용하면 `Hwp()` 생성자가 이 과정을 자동 처리한다.

한글 2024에서 `RegisterModule`을 호출했음에도 보안 팝업이 사라지지 않는 사례가 한컴 포럼에 보고되었는데, 대부분 레지스트리 경로 오류(HwpCtrl vs HwpAutomation)이거나 최신 패치 미적용이 원인이었다.

## 32비트·64비트 불일치와 다중 버전 충돌 해결법

한글은 전통적으로 32비트 애플리케이션이므로, **64비트 Python으로 32비트 한글에 연결하면 `com_error: Class not registered (-2147221164)` 에러**가 발생한다. 해결법은 단순하다—한글의 비트 수에 맞는 Python을 설치하면 된다. `struct.calcsize("P") * 8`로 현재 Python의 비트 수를 확인할 수 있다. 한컴오피스 2024부터는 64비트 설치 옵션이 존재하므로, 설치된 한글의 실제 비트 수를 먼저 확인해야 한다.

다중 버전 설치 충돌은 더 근본적인 문제다. `HWPFrame.HwpObject` ProgID는 **하나의 `LocalServer32` 경로만 가리킬 수 있으므로**, 마지막으로 설치·업데이트된 버전이 COM 서버를 독점한다. 특정 버전으로 전환하려면 관리자 권한 명령 프롬프트에서 해당 버전의 `hwp.exe -regserver`를 실행해야 한다:

```batch
cd "C:\Program Files (x86)\HNC\Office 130\HOffice130\Bin"  :: 한글 2024 예시
Hwp.exe -regserver
```

이론적으로 `HWPFrame.HwpObject.1`(구 버전)과 `HWPFrame.HwpObject.2`(신 버전)를 별도 CLSID에 연결하여 동시 사용이 가능하지만, 실제로는 두 ProgID가 같은 CLSID를 가리키는 경우가 대부분이므로 **레지스트리를 수동 편집하지 않는 한 신뢰하기 어렵다**. 런타임에 `hwp.Version` 속성으로 연결된 버전을 반드시 확인하는 것이 안전하다.

## 안정적 연결과 성능 최적화 실전 패턴

### Dispatch vs EnsureDispatch 선택 기준

`win32.Dispatch()`(late-binding)는 gen_py 캐시 문제가 없어 **가장 안정적**이지만 IDE 자동완성이 지원되지 않는다. `win32.gencache.EnsureDispatch()`(early-binding)는 자동완성이 가능하지만 캐시 파일 손상 시 `AttributeError: CLSIDToClassMap` 같은 에러가 발생한다. 캐시 문제가 생기면 `%LOCALAPPDATA%\Temp\gen_py` 폴더를 삭제하면 된다. 성능 차이는 미미하므로, **안정성 우선이면 Dispatch, 개발 편의성 우선이면 EnsureDispatch**를 선택한다.

### 권장 초기화·종료 시퀀스

```python
import win32com.client as win32

# 초기화
hwp = win32.Dispatch("HWPFrame.HwpObject")
hwp.RegisterModule("FilePathCheckDLL", "FilePathCheckerModule")
hwp.XHwpWindows.Item(0).Visible = True

# 작업 수행...

# 종료 (반드시 이 순서)
hwp.Clear(1)   # 문서 닫기 (1=변경사항 버림)
hwp.Quit()     # 한글 프로세스 종료
del hwp        # COM 참조 해제
```

**`Quit()` 미호출은 좀비 프로세스의 주범**이다. 작업 관리자에 hwp.exe가 다수 남아 메모리를 점유하는 현상은 거의 이 원인이다. 스레드 환경에서는 반드시 `pythoncom.CoInitialize()`/`CoUninitialize()` 쌍을 호출해야 `CoInitialize가 호출되지 않았습니다` 에러를 방지할 수 있다.

### 성능 최적화 핵심 세 가지

**첫째, 화면 갱신 끄기.** `hwp.XHwpWindows.Item(0).Visible = False`로 백그라운드 실행하면 대량 작업 시 **체감 10배 이상 속도 향상**이 가능하다.

**둘째, PutFieldText로 배치 입력.** 커서 이동 + InsertText 반복 대신, 한글 문서 템플릿에 누름틀(필드)을 배치하고 `hwp.PutFieldText("필드1\x02필드2", "값1\x02값2")`로 일괄 입력하면 훨씬 빠르다.

**셋째, CreateAction 캐싱.** `HAction.GetDefault/Execute`를 반복하기보다 `hwp.CreateAction("InsertText")`로 액션 객체를 한 번 생성하여 재사용하면 호출 오버헤드를 줄일 수 있다. `InitScan/GetText` 사용 후에는 **반드시 `ReleaseScan()`을 호출**해야 메모리 누수를 방지한다.

## Node.js·Electron에서의 COM 연결은 winax가 사실상 유일한 선택

Node.js에서 HWP COM 자동화에는 **winax**(npm) 패키지가 사실상 표준이다. `new winax.Object("HWPFrame.HwpObject")`로 COM 객체를 생성하며, Python의 win32com과 거의 동일한 API 패턴을 사용할 수 있다:

```javascript
const winax = require("winax");
const hwp = new winax.Object("HWPFrame.HwpObject");
hwp.XHwpWindows.Item(0).Visible = true;
hwp.Open("C:/path/test.hwp", "HWP", null);
```

Electron 환경에서는 **네이티브 모듈 리빌드가 필수**다. Electron의 V8 버전과 시스템 Node.js V8 버전이 다르기 때문에, `electron-rebuild` 또는 `npm rebuild winax --runtime=electron --target=<electron-version>`으로 정확한 타겟 버전에 맞춰 빌드해야 `NODE_MODULE_VERSION` 에러를 피할 수 있다. 또한 한글이 32비트 COM 서버인 경우 **Node.js도 x86 버전**을 사용해야 한다.

COM을 사용하지 않는 대안으로는 `node-hwp`(123jimin/node-hwp)가 있다. HWP 5.x 바이너리 파일을 직접 파싱하는 순수 Node.js 라이브러리로, 비 Windows 환경에서도 문서 읽기가 가능하다. Java 환경에서는 `hwplib`(neolord0/hwplib)이 유사한 역할을 한다.

## 한컴오피스 2024·NEX 시대의 자동화 방향

한컴오피스 2024는 COM API의 기본 구조(`HWPFrame.HwpObject`, Action/ParameterSet 체계)를 유지하면서도 세부 동작에서 여러 변경이 있었다. **HWPX(개방형 OWPML, 국가표준 KSX6101)**를 기본 저장 형식으로 권고하는 방향으로 전환 중이며, HWPX는 XML 기반이므로 COM 없이도 직접 파싱이 용이하다. COM 자동화 자체는 HWP·HWPX 모두 동일하게 작동한다.

웹 환경에서는 **웹한글 기안기(WebHwp)**가 ActiveX를 대체하는 공식 경로다. 서버 사이드에서 실행되므로 로컬 시스템 접근이 필요한 API는 제거되었고, 호출 방식도 JavaScript API로 변경되었다. 한컴오피스 2018 이상에서 최적의 호환성을 제공한다.

커뮤니티 차원에서는 **pyhwpx**(pip install pyhwpx)가 가장 활발한 Python 래퍼로, RegisterModule 자동 처리·고수준 필드 작업·상세 한국어 튜토리얼(wikidocs.net/book/8956)을 제공한다. 한글 2022 기준으로 개발되었으므로 2024 환경에서는 Visible 토글 등 수동 대응이 필요할 수 있다.

## 결론: 버전별 분기가 필수인 시대

한글 COM 자동화는 표면적으로 단순해 보이지만, **버전별 CLSID 차이, 한글 2024의 breaking change 3건, OCX 완전 제거, 32/64비트 불일치**라는 네 가지 함정이 실무에서 반복적으로 발생한다. 가장 견고한 접근법은 런타임에 `hwp.Version`을 확인하여 버전별 분기를 두고, `RegisterModule` 보안 설정을 정확히 수행하며, Dispatch/Quit/del 라이프사이클을 엄격히 관리하는 것이다. 한글 2024 이후로는 dynamic late-binding이 .NET 환경에서의 필수 패턴이 되었고, 장기적으로는 HWPX 직접 파싱과 WebHwp가 COM 의존성을 줄이는 방향이 될 것이다.
