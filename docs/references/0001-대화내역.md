COM 자동화를 통한 외부 프로세스 제어의 제한사항을 검토했다
COM 자동화를 통한 외부 프로세스 제어의 제한사항을 검토했다
네, 외부 프로세스에서 COM으로 조종하는 거 맞는데 실질적인 제한들이 꽤 있습니다.

구조 이해
COM Automation은 Out-of-process COM 서버 방식입니다. Electron이 COM 클라이언트가 되고, HWP/Excel이 별도 프로세스로 뜨는 구조예요. 프로세스 간 통신이라 기본적으로 느립니다.

주요 제한사항

1. HWP 보안 모듈 (한컴 특유)
   가장 큰 장벽입니다. 파일 열기/저장 전에 RegisterModule로 보안 DLL을 등록해야 하는데, 이 DLL이 한컴오피스 설치 경로에 종속됩니다. 경로가 다르면 직접 찾아야 하고, 버전마다 DLL 이름이나 인터페이스가 바뀌는 경우가 있어요.

2. 사용자 PC에 해당 프로그램이 설치되어 있어야 함
   당연한 말이지만, Wissly 입장에서는 "한컴/오피스 없으면 이 기능 비활성화" 처리를 해야 합니다. 설치 여부 감지는 레지스트리 조회로 가능합니다.

3. UI 스레드 문제
   COM 객체는 대부분 STA(Single-Threaded Apartment) 기반이라, 같은 스레드에서만 호출해야 합니다. Node.js에서 winax/node-win32ole을 쓸 때 멀티스레드 환경에서 크래시가 나는 이유가 이것입니다. Worker thread나 별도 프로세스로 격리하는 게 안전해요.

4. 모달 다이얼로그 블로킹
   자동화 중에 HWP/Excel이 사용자한테 다이얼로그를 띄우면 COM 호출이 블로킹됩니다. 예를 들어 저장 시 "덮어쓸까요?" 팝업이 뜨면 응답 대기 상태가 됩니다. Visible = false로 숨겨두거나, 저장 옵션을 명시적으로 지정해서 팝업을 억제해야 합니다.

5. 속도
   COM IPC는 메서드 호출 하나하나가 프로세스 간 마샬링을 거쳐서 호출당 수 ms~수십 ms 오버헤드가 있습니다. 셀 수천 개를 하나씩 쓰는 식이면 체감이 크게 느려집니다. 배열 단위로 묶어서 한 번에 넘기는 식으로 최적화가 필요해요.

6. 프로세스 정리 문제
   COM 클라이언트(Electron)가 크래시 나거나 참조를 제대로 해제 안 하면 HWP/Excel 프로세스가 좀비로 남습니다. 작업관리자에 숨어있는 채로요. Quit() 명시 호출 + try/finally 패턴이 필수입니다.

7. Node.js 네이티브 바인딩 빌드
   winax나 node-win32ole은 네이티브 .node 바인딩이라 VS Build Tools + node-gyp 빌드가 필요하고, Electron 버전과 Node ABI가 맞아야 합니다. Electron rebuild (electron-rebuild) 단계를 CI/빌드 파이프라인에 추가해야 해요. 이게 생각보다 골치 아픈 부분입니다.

Electron에서 현실적인 접근법
방법 장점 단점
winax / node-win32ole JS에서 직접 호출 빌드 복잡, STA 주의
Python child_process 안정적, 라이브러리 풍부 Python 설치 필요 or 번들링
PowerShell child_process 별도 설치 불필요 성능 더 느림

Worker Thread를 대체하는 게 아니라 child_process.fork를 대체하는 개념이에요.

포지셔닝
Worker Thread → 같은 프로세스, 공유 메모리, CPU 병렬 작업
child_process → 별도 프로세스, 격리, 근데 Electron 컨텍스트 없음
utilityProcess → 별도 프로세스, 격리, + Node.js + MessagePort 지원
child_process.fork랑 결정적으로 다른 점이 두 가지입니다.

첫째, 진입점 모듈이 반드시 패키지 내부 경로여야 합니다. child_process는 임의 경로에서 로드할 수 있지만 utilityProcess는 신뢰된 스크립트만 로드할 수 있도록 제한됩니다. 둘째, Renderer와 MessagePort로 직접 통신 채널을 열 수 있습니다.
